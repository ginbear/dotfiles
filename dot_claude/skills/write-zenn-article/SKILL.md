# Zenn 記事執筆スキル

Zenn の技術記事を執筆・レビューするスキル。

## 記事の配置先

```bash
# ghq 管理下の zenn リポジトリ
$(ghq root)/github.com/$(git config user.name)/zenn/articles/
```

## frontmatter

既存記事の frontmatter を参照してフォーマットを合わせる。`published: false` で作成し、公開はユーザーが判断する。ファイル名はケバブケースで内容を表すものにする。

## 文体のルール

### 基本方針

- 事実を中心に淡々と述べる
- 素直な文体にする。ビジネス書ライクな言い回しを避ける

### 避けるもの

- 太字の多用。原則使わない
- 「〜にもかかわらず、です。」のような回りくどい強調
- 「シンプルで」「パワフルな」「革新的な」などの修飾語
- 「〜してみませんか？」のような呼びかけ
- AI が書いた感のあるタイトルやフレーズ
- 箇条書きの乱用。文章で書けるなら文章にする

### 心がけるもの

- 既存記事のトーンに合わせる（`~/ghq/github.com/ginbear/zenn/articles/` を参照）
- 読者が持ち帰れる具体的なアクションや知見を含める
- 「いい話だね」で終わらない、実践的な内容にする

## 情報の取り扱い

### 公開してはいけない情報

記事に含める前に以下を確認し、該当するものはぼかす:

- サービス規模が特定される数字（Node 数、Pod 数、リクエスト数など）
- 具体的な環境名やバージョンの組み合わせ（環境構成が特定される）
- 本番環境での作業であることの明示（「あるアップグレードで」のようにぼかす）
- 社内のリポジトリ名、サービス名、ドメイン名
- AWS アカウント ID、リソース名、IP アドレス

該当する情報は具体的な値を避け、抽象的な表現に置き換える。

## レビューの観点

記事のレビューを求められた場合、以下の観点で確認する:

### 構成

- 読者が迷わない流れになっているか
- 同じことを複数箇所で言っていないか
- 見出しのレベル感が揃っているか

### 内容

- 主張に矛盾がないか
- 具体例が主張を裏付けているか
- 「慣れただけでは？」のような反論に答えられているか
- 読者が持ち帰れるアクションがあるか
- **技術的主張の検証**: ツールの挙動・仕様に関する記述は、公式ドキュメントまたは実際のコマンド出力で裏取りする。記憶だけで「〜という仕様」と書かない。未検証の場合はユーザーに明示する

### リスク

- 公開してはいけない情報が含まれていないか（上記チェックリスト参照）
- 炎上リスクのある表現がないか
  - 「手抜き」「省略」を正当化する表現
  - AI に任せすぎて安全性が疑われる表現
  - 過去の手順や他のチームの仕事を否定する表現
- 類似記事との差別化ができているか（WebSearch で確認）

## ワークフロー

### 1. 構成の相談

ユーザーと記事の方向性・構成を決める。いきなり書き始めない。

### 2. ドラフト作成

`published: false` で記事を作成する。

### 3. レビュー & 修正

上記のレビュー観点で確認し、指摘と修正を繰り返す。
ユーザーからの修正指示は次回以降の記事にも適用できるものであれば、このスキルに反映する。

### 4. タイトル候補

タイトルは複数候補を出してユーザーに選んでもらう。既存記事のタイトルを参考にする。
